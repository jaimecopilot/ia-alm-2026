# T8.1 – CATÁLOGO DE FALLOS TÍPICOS: ALUCINACIONES, DEPENDENCIAS FALSAS Y CÓDIGO INCORRECTO

A lo largo de este curso has experimentado el enorme potencial de las IAs generativas: generan código, planifican implementaciones, documentan, refactorizan y despliegan. Pero también has visto que no son infalibles. Las alucinaciones —cuando el modelo genera información plausible pero falsa— son el riesgo más insidioso porque el output parece correcto a simple vista. Comprender el catálogo completo de fallos típicos es el primer paso para mitigarlos.

Las alucinaciones de dependencias son el tipo más común y peligroso. La IA puede sugerir imports de librerías que no existen: from pandas_light import read_csv, import json_fix, from oracle_light import Connection. El nombre suena plausible —el modelo ha visto combinaciones de palabras similares en su entrenamiento— pero la librería no existe. La única defensa eficaz es verificar cada dependencia sugerida: buscar en PyPI, npm o el registro correspondiente antes de aceptar el import.

Las alucinaciones de API son más sutiles. La IA puede sugerir métodos que no existen en una librería real: json.auto_fix(), requests.get().parse_json(), datetime.now().to_iso(). El método parece lógico pero no forma parte de la API real. La verificación contra la documentación oficial de la librería es obligatoria.

El código funcionalmente incorrecto es quizás el fallo más peligroso porque no produce errores evidentes. La IA puede generar una función de validación de emails que use una expresión regular incorrecta, aceptando emails inválidos o rechazando válidos. Puede implementar una lógica de solapamiento de horarios que funcione para el caso feliz pero falle en casos límite. Solo los tests exhaustivos pueden detectar estos fallos.

Las dependencias falsas en documentación son otro tipo de alucinación. La IA puede generar un README o un ADR que haga referencia a tecnologías, patrones o decisiones que nunca se implementaron. Si no se revisa, puede inducir a error a futuros desarrolladores.

Los sesgos en la generación de código son un riesgo más sutil. La IA puede favorecer ciertos frameworks o librerías no porque sean los mejores para tu caso, sino porque están sobrerrepresentados en sus datos de entrenamiento. También puede generar ejemplos con nombres anglosajones, asumir formatos de fecha estadounidenses o validar nombres solo con caracteres ASCII.

El código con vulnerabilidades de seguridad es un riesgo crítico. La IA aprende de código público que contiene malas prácticas: consultas SQL concatenadas, contraseñas hardcodeadas, falta de sanitización de entradas. No distingue entre código seguro e inseguro. La revisión de seguridad es obligatoria.

La desalineación con los requisitos es un riesgo de proceso. La IA puede generar código que implementa una funcionalidad no solicitada, omite una validación crítica o malinterpreta un criterio de aceptación ambiguo. Este riesgo se mitiga escribiendo especificaciones claras con criterios Gherkin y revisando el código generado contra los requisitos originales.
